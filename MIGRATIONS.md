## Migration Workflow

### How Alembic Works (High Level)

```
SQLAlchemy Models → env.py → Compares with DB Schema → Generates Migration Script → Applied to DB
```

**Key Files:**
- `alembic/env.py` - Configuration (reads models, compares changes)
- `alembic/script.py.mako` - Template for new migration files
- `alembic/versions/` - All migration files (version controlled)
- `alembic/alembic_version` - Tracks which migration was last applied

## Development: Creating Migrations

### ✅ Workflow for Adding New Tables/Columns

#### 1. Create SQLAlchemy Model

Create a new model file in `app/models/`. Example: `app/models/player.py`

```python
from sqlalchemy import Column, Integer, String, DateTime, ForeignKey
from sqlalchemy.sql import func
from app.db import Base

class Player(Base):
    __tablename__ = "players"
    
    id = Column(Integer, primary_key=True, index=True)
    name = Column(String(255), nullable=False)
    team_id = Column(Integer, ForeignKey("teams.team_id"), nullable=False)
    position = Column(String(50), nullable=True)
    created_at = Column(DateTime(timezone=True), server_default=func.now())
    updated_at = Column(DateTime(timezone=True), onupdate=func.now())
```

#### 2. Import Model in `app/models/__init__.py`

```python
from app.models.player import Player  # Add this line
```

#### 3. Generate Migration File (Autogenerate)

```bash
# From the project root directory
alembic revision --autogenerate -m "add players table"
```

**Expected Output:**
```
  Generating .../alembic/versions/0abc1234567f_add_players_table.py ... done
```

#### 4. Review Generated Migration

Check the file: `alembic/versions/0abc1234567f_add_players_table.py`

```python
def upgrade():
    # Should contain: op.create_table('players', ...)
    pass

def downgrade():
    # Should contain: op.drop_table('players')
    pass
```

⚠️ **Important**: Always review autogenerated migrations before applying!

#### 5. Apply Migration Locally (Test)

```bash
# Apply the latest migration
alembic upgrade head

# Or apply specific migration
alembic upgrade 0abc1234567f
```

**Verify in database:**
```bash
psql postgresql://fover_user:242374@localhost:5432/fover_db
\dt  # List all tables
```

#### 6. Commit to Version Control

```bash
git add alembic/versions/0abc1234567f_add_players_table.py
git add app/models/player.py
git add app/models/__init__.py
git commit -m "feat: add players table with migration"
```

### Modification Examples

#### Example A: Add a New Column to Existing Table

**Change in Model:**
```python
class Player(Base):
    __tablename__ = "players"
    # ... existing columns ...
    jersey_number = Column(Integer, nullable=True)  # NEW COLUMN
```

**Generate Migration:**
```bash
alembic revision --autogenerate -m "add jersey_number to players"
```

**Result (auto-generated):**
```python
def upgrade():
    op.add_column('players', sa.Column('jersey_number', sa.Integer(), nullable=True))

def downgrade():
    op.drop_column('players', 'jersey_number')
```

#### Example B: Add Unique Constraint

**Change in Model:**
```python
class Player(Base):
    __tablename__ = "players"
    __table_args__ = (
        UniqueConstraint('team_id', 'jersey_number', name='uq_team_jersey'),
    )
```

**Generate & Apply:**
```bash
alembic revision --autogenerate -m "add unique constraint team_id_jersey_number"
alembic upgrade head
```

#### Example C: Add Index

**Change in Model:**
```python
class Player(Base):
    name = Column(String(255), nullable=False, index=True)  # Added index=True
```

**Generate & Apply:**
```bash
alembic revision --autogenerate -m "add index on players.name"
alembic upgrade head
```

## Production: Deploying Migrations

### 🚀 Before Deployment Checklist

- [ ] All migrations applied locally: `alembic current` shows latest version
- [ ] Database backup created
- [ ] Migration files are reviewed and tested
- [ ] Team informed of planned downtime (if needed)
- [ ] Rollback plan prepared

### Safe Deployment Steps

#### Step 1: Review Migration Status

```bash
# Show current migration version
alembic current

# Show latest available migration
alembic heads

# Show all migrations (with status)
alembic history --verbose
```

#### Step 2: Deploy to Production VPS

**Option A: Direct SSH (Recommended for small deployments)**

```bash
# SSH into production server
ssh user@production-server

# Navigate to project
cd /path/to/fover_backend

# Set production database URL
export DATABASE_URL="postgresql://prod_user:secure_pass@prod-db:5432/fover_prod"

# Apply migrations
alembic upgrade head

# Verify
alembic current
```

**Option B: Docker/Kubernetes**

Add to `Dockerfile` or deployment script:

```dockerfile
# Run migrations before starting app
CMD alembic upgrade head && gunicorn app.main:app
```

Or as a separate init container in Kubernetes:

```yaml
initContainers:
  - name: db-migration
    image: your-registry/fover-backend:latest
    command: 
      - alembic
      - upgrade
      - head
    env:
      - name: DATABASE_URL
        valueFrom:
          secretKeyRef:
            name: db-credentials
            key: url
```

#### Step 3: Verify Production

```bash
# Connect to production database
psql $DATABASE_URL

# Check tables and schema
\dt

# Exit
\q
```

## Troubleshooting

### ❌ Problem: "No changes detected in schema"

When running `alembic revision --autogenerate -m "message"`, no migration is created.

**Causes & Solutions:**

1. **Model not imported in `env.py`**
   - Add import: `from app.models import MyNewModel`
   - Restart terminal and try again

2. **Model not imported in `app/models/__init__.py`**
   - Add: `from app.models.my_model import MyNewModel`

3. **Column changes are cosmetic**
   - Alembic only detects structural changes, not docstring changes

**Fix:**
```bash
# Reset Python environment
deactivate && source venv/bin/activate  # Linux/Mac
# or on Windows:
venv\Scripts\deactivate && venv\Scripts\activate

# Try again
alembic revision --autogenerate -m "your message"
```

---

### ❌ Problem: "Can't locate revision identified by 'abc123'"

**Cause:** Migration file is missing from `alembic/versions/`

**Solution:**
```bash
# Check revision chain
alembic history --verbose

# Identify the issue and either:
# - Restore the missing migration file from git
# - Or manually fix the alembic_version in database
```

---

### ❌ Problem: "Can't drop column - foreign key constraint"

**Cause:** Dropping a column that's referenced by another table

**Solution:** Manually edit the migration to drop constraints first:

```python
def upgrade():
    # Drop foreign keys first
    op.drop_constraint('fk_other_table_column_id', 'other_table')
    # Then drop the column
    op.drop_column('my_table', 'my_column')

def downgrade():
    # Recreate in reverse order
    op.add_column('my_table', sa.Column('my_column', sa.Integer()))
    op.create_foreign_key('fk_other_table_column_id', 'other_table', 
                          'my_table', 'my_column', 'id')
```

---

### ❌ Problem: "Can't modify column type"

PostgreSQL is strict about type changes when data exists.

**Solution:** Use the `--render-as-batch` flag (already set in env.py) or manually edit migration:

```python
def upgrade():
    # For large tables with data, use a 2-step approach:
    op.alter_column('my_table', 'my_column',
                    type_=sa.String(500),
                    existing_type=sa.String(255))
```

## Safety Best Practices

### ✅ DO:

1. **Always backup production database before migrations**
   ```bash
   pg_dump $DATABASE_URL > backup_$(date +%Y%m%d_%H%M%S).sql
   ```

2. **Test migrations in a staging environment first**
   ```bash
   # Clone production database locally or to staging
   alembic upgrade head
   # Test your app
   ```

3. **Use descriptive migration names**
   ```bash
   # Good
   alembic revision --autogenerate -m "add match_rating_column_to_matches"
   
   # Bad
   alembic revision --autogenerate -m "update"
   ```

4. **Review every generated migration**
   - Open the file and ensure changes are expected
   - Verify the downgrade path makes sense

5. **Version control all migrations**
   ```bash
   git add alembic/versions/
   git commit -m "feat: new migration"
   git push
   ```

6. **Implement gradual rollout for large migrations**
   - Deploy to dev → staging → production with time gaps
   - Monitor application logs for errors

### ❌ DON'T:

1. **Don't manually edit the database schema**
   - Always use migrations
   - Alembic won't detect manual changes

2. **Don't run migrations without a backup**
   ```bash
   # Bad
   alembic upgrade head
   
   # Good
   pg_dump $DATABASE_URL > backup.sql && alembic upgrade head
   ```

3. **Don't commit incomplete migrations**
   - All migrations must be working and tested

4. **Don't delete migration files**
   - Even old ones - they're part of version history
   - Mark as invalid instead if needed

5. **Don't apply migrations to production without staging test**
   - Always test in non-production environment first

## Commands Reference

### Quick Reference Table

| Task | Command |
|------|---------|
| Check current version | `alembic current` |
| Show all migrations | `alembic history --verbose` |
| Generate migration | `alembic revision --autogenerate -m "description"` |
| Apply all pending | `alembic upgrade head` |
| Apply specific version | `alembic upgrade 0abc1234567f` |
| Rollback 1 version | `alembic downgrade -1` |
| Rollback to version | `alembic downgrade 0abc1234567f` |
| Downgrade to base | `alembic downgrade base` |
| Merge branches | `alembic merge heads -m "merge"` |

## Integration with CI/CD

### GitHub Actions Example

```yaml
name: Database Migrations

on: [pull_request, push]

jobs:
  migrate:
    runs-on: ubuntu-latest
    services:
      postgres:
        image: postgres:15
        env:
          POSTGRES_USER: fover_user
          POSTGRES_PASSWORD: password
          POSTGRES_DB: fover_db
        options: >-
          --health-cmd pg_isready
          --health-interval 10s
          --health-timeout 5s
          --health-retries 5

    steps:
      - uses: actions/checkout@v3
      - uses: actions/setup-python@v4
        with:
          python-version: '3.11'
      - run: pip install -r requirements.txt
      - run: |
          export DATABASE_URL=postgresql://fover_user:password@localhost:5432/fover_db
          alembic upgrade head
      - run: alembic current
```
